ℹ️ How the exam works

Mechanics, scoring, scenarios — from the official Certification Exam Guide and candidate field reports.

FormatStructure, scoring & logistics
▶
  • 60 multiple-choice questions in 120 minutes — one correct answer + three distractors per question. No penalty for wrong answers: always answer; blanks score the same as wrong.
  • Scaled score 100–1,000, passing threshold 720. Pass/fail result.
  • ~$99 per attempt. Online proctored, closed-book: webcam + screen-share monitoring, no Claude, no docs, no external tabs.
  • Distractors are designed for candidates with incomplete knowledge: plausible, often technically true — but not the «best» answer.
  • Target candidate: solution architect with 6+ months of hands-on practice with the Claude API, Agent SDK, Claude Code and MCP.

⚠️ Reality checkYou cannot memorize your way through
▶

The practice exam is recognition-based («what is X?»); the live exam is judgment-based («your system is failing — which architecture fixes it?»). One documented candidate scored 980/1000 on practice and 744/1000 live in the same week — a 236-point swing measuring the same knowledge.

"The practice exam asks you to recognize a concept. The live exam drops you into a broken production system and asks you to repair it."
  • Treat a 900+ practice score as a floor to clear, not a forecast — the practice exam is calibrated easier.
  • Passers report almost no rote-recall questions (e.g. no stop_reason enumeration) — the exam tests behavioral API judgment: when a rolling context window helps, when the Batch API is the wrong tradeoff despite the discount, how statelessness forces the application layer to own continuity.
  • Study method: convert every concept into a production-failure → decision pair, not a definition.

⚠️ Reality checkThe 60/40 problem — and the hidden safety topic
▶

Candidate reports estimate only ~60% of live exam content maps to the official study guide and practice materials; the other ~40% draws on Anthropic Academy courseware and Cookbook notebooks that practice barely touches. The neglected clusters:

  • Structured output & tool design (deeper than the practice set — see D4).

field reportGolden Rule 1
Self-test90-second readiness diagnostic
▶

Six failure scenarios — if you can't name the fix instantly, that's your weak area:

  1. Endless agent loop → exit on stop_reason, never iteration caps or output-text scanning. (D1)
  2. Tool failures → return explicit, structured errors the model can act on — never empty results or fake success. (D2)
  3. Configuration conflicts → deeper-scoped rules and higher-level rules combine and get interpreted together — they don't silently override each other. (D3)
  4. Hallucinated extraction → the schema must allow null / explicit "unclear" values — never force the model to invent or omit. (D4)
  5. Lost context → restructure so critical facts sit at context edges / in persistent blocks — never assume the API remembers. (D5)
  6. Injection vulnerability → gate privileged actions in code; isolate untrusted input — never a disclaimer or prompt strengthening. (Safety)

D1 — Agentic Architecture & Orchestration

27%

The heaviest domain (~16 of 60 questions) and the most "bankable" one: concrete vocabulary, learnable patterns. Agentic loop, coordinator/subagents, hooks, decomposition, sessions.

Key concepts
Agentic loopstop_reason drives the loop
▶

The loop is a while driven by the response's stop_reason field: tool_use → execute the requested tools, append the results to the conversation history and call the API again; end_turn → the turn is complete, stop. Also exists: pause_turn (server-side iteration limit for server tools, e.g. web search: re-send and the server resumes) and refusal.

Which tool to call next is a model-driven decision (the model reasons over context), not a pre-configured decision tree or fixed tool sequence.

Q58Task 1.1✓ Messages API docs
Agentic loopTermination anti-patterns
▶

Three mistakes the exam punishes:

  • Parsing the output text to decide if it's done («looks finished») — unreliable.
  • Arbitrary iteration cap as the primary stopping mechanism (fine as a safety net, wrong as the logic) — it masks the symptom instead of fixing the loop.
  • Checking for assistant text as a completion indicator.

The only correct signal is stop_reason.

Task 1.1
Multi-agentCoordinator as the central hub
▶

Hub-and-spoke topology: the coordinator observes all interactions, handles errors uniformly, and decides what information each subagent receives. Subagents never talk to each other directly. The coordinator owns: decomposition, delegation, aggregation, and it dynamically selects which subagents to invoke based on query complexity (not always the full pipeline).

Iterative refinement: the coordinator evaluates the synthesis output for gaps, re-delegates targeted queries, and re-invokes synthesis until coverage is sufficient.

Q2Q8Task 1.2
SubagentsIsolated context — pass it explicitly, and only what's needed
▶

«A subagent inherits nothing. It is a fresh context.» Subagents don't get the coordinator's conversation history and don't share memory between invocations. All relevant context (e.g. web-search results + document analysis for the synthesis agent) must be included in the subagent's prompt, in a structured format that separates content from metadata (URLs, document names, page numbers) to preserve attribution.

But scoping cuts both ways: each subagent should receive a tight, purpose-built prompt plus least-privilege tools — sharing the full conversation history creates privacy, cost and quality problems (see the trap below).

Task 1.3
SubagentsTask tool, allowedTools, parallel spawning
▶
  • Subagents are spawned via the Task tool; the coordinator can only invoke them if its allowedTools includes "Task".
  • Each subagent type is defined by an AgentDefinition: description, system prompt, tool restrictions.
  • Parallel = multiple Task calls in the same coordinator response; separate turns = sequential.
  • Coordinator prompts should specify research goals and quality criteria, not step-by-step procedures — leave subagents room to adapt.
Task 1.3
StateMoving state between agents: three mechanisms
▶

State transfer is always deliberate. Match the mechanism to the data:

  • Small inputs → a structured message inside the spawning prompt.
  • Large artifacts → shared files (avoids prompt bloat).
  • Findings → machine-readable JSON in tool-call results.
field reportTask 1.3
APIThe Claude API is stateless
▶

«Each call is independent, and the model retains nothing between calls.» This single fact explains several exam answers:

  • The full message history must be re-sent on every request for conversational coherence.
  • Subagents need explicit context passing — there is nothing to inherit.
  • The application layer owns continuity: session storage, summaries, case-facts blocks — never «the API remembers».
Task 5.1field report
EnforcementHooks: deterministic guarantees
▶

Two Agent SDK hook patterns:

  • Intercept the outgoing tool call to block policy-violating actions (e.g. refund > $500 → block and redirect to human escalation). A $500 limit belongs in a PreToolUse hook, not the system prompt: «the model cannot talk its way around a hook, because the hook runs in your code, not in the model's reasoning.»
  • PostToolUse: intercept results and transform them before the model processes them (e.g. normalize Unix vs ISO 8601 timestamps, numeric vs string codes from third-party MCP tools).

Rule: hooks for «must» (guaranteed compliance), prompts for «should» (probabilistic).

Q59Task 1.5
DecompositionPrompt chaining vs dynamic decomposition
▶
  • Prompt chaining (fixed sequential pipeline) for predictable multi-aspect workflows — e.g. review: analyze each file individually, then a cross-file integration pass.
  • Dynamic/adaptive decomposition for open-ended investigations — e.g. «add tests to a legacy codebase»: first map the structure, identify high-impact areas, then a prioritized plan that adapts as dependencies are discovered.
Task 1.6
Sessions--resume, fork_session, stale context
▶
  • --resume <name> resumes a named session to continue an investigation across work sessions.
  • fork_session creates independent branches from a shared baseline — e.g. compare two refactoring strategies from the same codebase analysis.
  • If the old session's tool results are stale (the code changed a lot), starting a fresh session with a structured summary is more reliable than resuming.
  • Resuming after limited changes? Tell the session which files changed for targeted re-analysis instead of full re-exploration.
Task 1.7
Exam traps
⚠️ TrapPartial coverage? Blame the decomposition
▶

If the final result covers only part of the requested domain (only visual arts instead of music+writing+film) and every subagent executed its own piece correctly, the root cause is the coordinator's overly narrow task decomposition — not the subagents, not the search queries, not the relevance filters.

Q4Sample Q7
⚠️ TrapThe coordinator that shared too much
▶

Watch for scenarios where the coordinator passes the full conversation history (or full customer records) to every subagent: the symptoms are data leaks into outputs, token-cost explosions, or subagents returning unfocused answers. The fix is re-scoping the decomposition — each subagent gets only the context its task needs — not «strengthening» the subagents' prompts.

field reportTask 1.3
⚠️ TrapExplicit partitioning, not post-hoc dedup
▶

To stop two agents from investigating the same things (double tokens, zero coverage gain), the coordinator must explicitly partition the research space before delegating (distinct subtopics or source types) — not let them run in parallel and deduplicate afterwards.

Q11
⚠️ TrapErrors: local recovery, escalation with context
▶

Handle the error at the lowest level that can resolve it: the subagent retries locally once or twice for transient failures; it escalates to the coordinator only what it cannot resolve, with complete structured context: failure type, attempted query, partial results, alternatives tried. Never: a generic status («search unavailable»), an error disguised as success (empty result set marked ok), an exception that kills the whole workflow.

SDK detail: return "is_error": true in the tool result to keep the loop alive — an uncaught exception kills the entire query call.

Q3Q6Q9Sample Q8
⚠️ Trap«0 results» ≠ timeout
▶

«0 results» is a valid outcome: the query succeeded, nothing matched. A timeout/access failure instead requires a retry decision. Confusing the two leads to pointless retries or — worse — discarding a legitimate result.

Q12Task 2.2
⚠️ TrapMulti-issue request → decompose and parallelize
▶

«Billed twice + discount didn't apply + I want to cancel»: if the agent investigates sequentially, re-fetching customer data per issue (12+ tool calls, 54% success), the fix is to decompose the request into distinct issues and investigate them in parallel with shared customer context, then synthesize one resolution. Not: verification checkpoints, one mega-tool, few-shot examples of ideal sequences.

D2 — Tool Design & MCP Integration

18%

Descriptions, structured errors, tool distribution, tool_choice, MCP, built-in tools.

Key concepts
Tool designThe description is the primary routing mechanism
▶

Not the name, not the system prompt: the model selects tools based on the description. A good description includes: accepted input formats, example queries, edge cases, and explicit boundaries («use for X, not for Y — for Y use tool Z»).

If a generic tool keeps being misused, also consider splitting it into purpose-specific tools: analyze_document → extract_data_points, summarize_content, verify_claim_against_source.

Q46Q57Q60Task 2.1
MCP errorsStructured errors: category + retryability
▶

An MCP tool communicates failures with the isError flag and structured metadata:

  • errorCategory: transient (timeout, service down) / validation (invalid input) / business (policy violation) / permission.
  • isRetryable: prevents wasted retries on unrecoverable errors.
  • Human-readable description; for business errors also a customer-friendly explanation with retriable: false.

A generic «Operation failed» blocks any intelligent recovery decision by the agent.

Task 2.2
DistributionFew tools, the right ones: 4-5 per role, not 18
▶

Too many tools degrade selection reliability (more decision complexity). An agent holding tools outside its specialization tends to misuse them (a synthesis agent doing web searches). Rules:

  • Each subagent gets only its role's tools.
  • For high-frequency cross-role needs: a scoped, limited tool (e.g. verify_fact for the synthesis agent → handles the 85% of simple fact-checks; the complex 15% stays with the coordinator).
  • Replace generic tools with constrained alternatives: fetch_url → load_document that validates formats at the interface level.
Q10Q15Task 2.3
APItool_choice: auto / any / forced
▶
  • "auto" (default): the model may call a tool or answer with text.
  • "any": it must call a tool (its choice which) — guarantees structured output when multiple extraction schemas exist and the document type is unknown.
  • {"type": "tool", "name": "..."} (forced): calls that tool — e.g. force extract_metadata before enrichment steps, which continue in follow-up turns.
Task 2.3Task 4.3
MCPConfig scoping and credentials
▶
  • Project .mcp.json = team tooling, version-controlled, single source of truth. Credentials via environment variable expansion (${GITHUB_TOKEN}): the file gets committed, the secrets don't.
  • User-level ~/.claude.json = personal/experimental servers.
  • Tools from all configured servers are discovered at connection time and available simultaneously.
  • Prefer community MCP servers for standard integrations (Jira); custom servers only for team-specific workflows.
Q44Task 2.4
MCPResources and «competitive» descriptions
▶
  • MCP resources expose content catalogs (DB schemas, documentation hierarchies, issue summaries) giving the agent visibility without exploratory tool calls.
  • If the agent prefers built-in tools (Grep) over more capable MCP tools, the cause is a poor MCP description: enrich it with detailed capabilities and outputs.
Task 2.4
Built-in toolsGrep vs Glob vs Read/Write vs Edit
▶
  • Grep = search file contents (function names, error messages, imports).
  • Glob = find files by path/name pattern (**/*.test.tsx).
  • Read/Write = whole-file operations; Edit = targeted change with a unique text match. If Edit fails on a non-unique match → fallback Read + Write.
  • Incremental exploration: Grep to find entry points → Read to follow imports and trace flows. Don't read every file upfront. To trace usage through wrapper modules: first list all exported names, then search each name across the codebase.
Task 2.5
Exam traps
⚠️ TrapWrong tool picked? Fix the description first, not few-shot
▶

When an agent picks the wrong tool, the temptation is to add few-shot examples to the prompt. But if the cause is an ambiguous/minimal description, the correct fix (and the highest-leverage «first step») is the tool description. Few-shot is the second tier. Routing layers, classifiers, and tool consolidation are over-engineering distractors for a «first step».

Q46Q57Q60Sample Q2
⚠️ TrapSemantic overlap → rename, don't just expand
▶

Fine distinction: if descriptions are minimal → expand them (Q57). If two tools have near-identical names+descriptions (analyze_content vs analyze_document, «analyzes content/documents and extracts key information»), the right answer is to rename the ambiguous tool (extract_web_results) and rewrite its description specifically — it removes the overlap at the root.

Q7Task 2.1
⚠️ TrapKeyword-driven routing? Check the system prompt
▶

If behavior varies systematically with one word («account» in the message → 78% get_customer; without → 93% lookup_order) and the descriptions are already clear, the root cause is a keyword-sensitive instruction in the system prompt creating unintended associations — not the model's training, not missing negative examples.

Q56Task 2.1
⚠️ TrapHeterogeneous formats → PostToolUse hook
▶

Different tools (especially unmodifiable third-party MCP) returning different formats (Unix vs ISO 8601 timestamps, numeric vs string codes)? A PostToolUse hook normalizes deterministically before the model interprets the data — more reliable than documenting the formats in the prompt.

Q59Task 1.5
⚠️ TrapSequential tool calls → prompt for bundling
▶

If the agent calls get_customer and lookup_order in separate turns even when both are needed (4+ loops per resolution), the simplest fix is to instruct it in the prompt to bundle tool requests into a single turn — it leverages the native ability to request multiple tools at once. Distractors: speculative execution, raising max_tokens, composite tools (valid but more invasive).

D3 — Claude Code Configuration & Workflows

20%

CLAUDE.md, rules, skills, slash commands, plan mode, iterative refinement, CI/CD.

Key concepts
ConfigCLAUDE.md hierarchy + @import + /memory
▶
  • Three levels: user (~/.claude/CLAUDE.md, personal, NOT shared via git), project (.claude/CLAUDE.md or root, version-controlled = the team's source of truth), directory (CLAUDE.md in subfolders).
  • Levels combine and get interpreted together — deeper-scoped rules don't silently override higher-level ones; conflicts must be resolved by editing, not assumed away.
  • @import references external files to keep CLAUDE.md modular — each package imports only the standards it needs.
  • /memory shows which memory files are loaded — the diagnostic command when behavior is inconsistent across sessions.
Q37Task 3.1
ConfigCLAUDE.md vs Skills vs .claude/rules/
▶
  • CLAUDE.md = always in memory in every session → only universal standards (coding style, testing).
  • Skills = loaded on-demand when Claude recognizes a trigger in the description → workflow-specific guidance (PR review, deploy, migration).
  • .claude/rules/ = files with YAML frontmatter and a paths glob field → activate only for matching file paths. Better than directory-level CLAUDE.md when the convention applies to files scattered across the codebase (e.g. **/*.test.tsx).
Q33Q38Q40Q42Sample Q6
SkillsFrontmatter: fork, allowed-tools, argument-hint
▶
  • context: fork: the skill runs as an isolated subagent with its own context window — verbose output (scanning hundreds of files) doesn't pollute the main conversation, which only receives a summary. The skill becomes the subagent's prompt and does not see the main history. Combine with agent: Explore / agent: Plan.
  • allowed-tools: restricts tools during execution (prevents destructive actions).
  • argument-hint: enforces required parameters when the skill is invoked without arguments.
  • Three different problems = three distinct frontmatter fixes, combinable in the same file.
Q35Q39Q43Q45✓ code.claude.com/docs
SkillsPrecedence and sharing
▶
  • A personal skill (~/.claude/skills/commit/) with the same name overrides the project one — individual customization without touching the team's workflow.
  • Team slash commands → .claude/commands/ in the repository (version-controlled, available on every clone/pull); ~/.claude/commands/ is personal.
Q36Q41Sample Q4
WorkflowPlan mode vs direct execution
▶
  • Plan mode: large-scale changes, multiple valid approaches, architectural decisions, multi-file work (monolith→microservices, a library migration across 45+ files, choosing webhook/bot/App). Explore and design before touching code — prevents costly rework.
  • Direct execution: well-scoped, clear changes (a single-file bug fix with a stack trace, adding a validation).
  • Combine them: plan to investigate, direct to implement the plan. The Explore subagent isolates verbose discovery and returns summaries.
Q32Q34Sample Q5
RefinementIteration: examples, test-driven, interview
▶
  • 2-3 concrete input→output examples when prose descriptions get interpreted inconsistently.
  • Test-driven iteration: write the test suite first (expected behavior, edge cases, performance), then iterate by sharing test failures.
  • Interview pattern: have Claude ask questions before implementing, to surface considerations you didn't anticipate (cache invalidation, failure modes) in unfamiliar domains.
  • Interacting problems → all in one detailed message; independent problems → sequential fixes.
Q31Task 3.5
CI/CDHeadless, structured output, context
▶
  • claude -p "..." (--print) = non-interactive mode: prints and exits. --batch and CLAUDE_HEADLESS do not exist.
  • --output-format json + --json-schema for machine-parseable findings (e.g. inline PR comments) — CLI-level enforcement, not a «parseable format» requested in the prompt.
  • CLAUDE.md is the context channel for CI: testing standards, review criteria, available fixtures — improves generated-test quality and reduces low-value output.
Q16Q26Sample Q10✓ Structured outputs
Exam traps
⚠️ TrapRule followed only by some? It's user-level
▶

If only some developers follow a rule and new team members don't, the most likely cause is that it lives in the author's personal ~/.claude/CLAUDE.md instead of the project's .claude/CLAUDE.md. Fix: move it to project level (version-controlled, shared).

Q37
⚠️ TrapDuplicate test suggestions? Missing context
▶

If 6 of 10 suggested tests duplicate scenarios already covered by the existing suite, the root cause is that Claude doesn't know which tests exist. Fix: include the existing test file in context. Distractors: asking for fewer suggestions, «focus on edge cases», keyword-based post-filtering.

Q24Task 3.6
⚠️ TrapDuplicate review comments after new commits
▶

Re-running review after fixes → comments duplicating findings already addressed. Fix: include the previous findings in context and instruct Claude to report only new or still-unresolved issues. Post-processing by file-path/description matching is the distractor (fragile); skipping intermediate commits or narrowing scope reduces thoroughness.

Q25Task 3.6
⚠️ TrapAmbiguous/architectural task → plan mode from the start
▶

If the complexity is already stated in the requirements (dozens of files, service boundaries, multiple valid approaches), the answer is enter plan mode from the start — not «direct execution, switch to plan if I hit complexity» (the complexity is already known), not hyper-detailed upfront instructions (they presume you already know the right structure).

D4 — Prompt Engineering & Structured Output

20%

Explicit criteria, few-shot, JSON schema via tool use, validation & retry, batch, multi-instance review.

Key concepts
PrecisionExplicit criteria > vague instructions
▶

«Check that comments are accurate» → inconsistent results. «Flag a comment only if the claimed behavior contradicts the actual code behavior» → a verifiable criterion. Define what to report (bugs, security) and what to skip (minor style) with concrete examples per severity level.

⚠️ «Be conservative» / «only high-confidence findings» do not improve precision — they're the classic distractors.

Q22Task 4.1
Few-shotWhen few-shot is the right answer
▶

2-4 targeted examples showing the why of one choice over plausible alternatives. They work for:

  • Consistent output format (location, issue, severity, suggested fix).
  • Handling ambiguous cases (tool selection, escalation criteria) — they generalize to novel patterns.
  • Reducing extraction hallucinations (informal measurements, varied document structures, empty/null fields).

They're a distractor when the problem is structural: an ambiguous description, information missing from context, a constraint that must be programmatic.

Q20Q47Q49Task 4.2
Structured outputTool use + JSON schema = guaranteed syntax
▶

Defining an extraction tool with the JSON schema as its input and extracting data from the tool_use block is the most reliable approach to schema-compliant output: it eliminates JSON syntax errors. It does not eliminate semantic errors: line items that don't sum to the total, values in the wrong field — those need downstream validation.

Multiple possible schemas and unknown document type → tool_choice: "any". To guarantee one specific extraction before other steps → forced {"type":"tool","name":"extract_metadata"}.

Task 4.3✓ Structured outputs
Structured outputAnti-fabrication schema design
▶
  • Optional fields when the document may not contain the information — a required field pushes the model to fabricate values just to satisfy the schema. The schema must allow null / explicit absence, never force invention or silent omission.
  • Enums with "unclear" for ambiguous cases and "other" + a detail string for extensible categories.
  • Format-normalization rules in the prompt alongside the schema (inconsistently formatted sources).
Task 4.3field report
ValidationRetry with error feedback — and its limits
▶

On failed validation: a follow-up request with original document + failed extraction + specific errors → the model self-corrects. Works for format/structure errors.

Retry is useless when the information isn't in the source (e.g. it lives in an external document that wasn't provided) — recognizing this case is tested.

Self-validation patterns: extract calculated_total alongside stated_total to flag discrepancies; a conflict_detected boolean; a detected_pattern field on findings to analyze recurring false positives.

Task 4.4
BatchMessage Batches API: when and how
▶
  • 50% cost savings, up to 24h, no latency SLA → good for latency-tolerant workloads (overnight reports, weekly audits, nightly test generation); never for blocking workflows (pre-merge checks).
  • No multi-turn tool use inside a request: it cannot execute a tool mid-request and let Claude continue.
  • custom_id correlates requests and responses (results come back unordered).
  • Submission frequency computed from SLAs: 4-hour windows to guarantee a 30-hour SLA with 24-hour processing.
  • Failures: resubmit only the failed items (identified via custom_id) with fixes (e.g. chunking documents over the context limit). Refine the prompt on a sample before full volume.
Q18Q19Q21Q30Sample Q11Task 4.5
ReviewMulti-instance and multi-pass
▶
  • Self-review is weak by construction: the model keeps its generation reasoning and won't question its own decisions. An independent instance without that context («fresh eyes») beats both self-review instructions and extended thinking.
  • Multi-pass: per-file for local issues + an integration pass for cross-file flows — against attention dilution (uneven feedback, obvious bugs missed, contradictory judgments on the same pattern).
  • Self-reported confidence alongside each finding for calibrated review routing.
Q17Q27Sample Q12Task 4.6
Exam traps
⚠️ TrapRequirements misread → examples, not more prose
▶

When prose requirements get interpreted differently on every iteration (transformations with ambiguous nesting/format), 2-3 concrete input→output examples remove the ambiguity better than more detailed instructions or the JSON schema alone.

Q31
⚠️ TrapTrust eroded by false positives → disable noisy categories
▶

FP rate by category: security 8%, performance 18%, style 52%, docs 48% → developers dismiss everything unread. Fix: temporarily disable the noisy categories, keep the precise ones, improve the prompts, then re-enable. Distractors: visible confidence scores (doesn't stop the erosion), few-shot «over the next few weeks» (too slow), uniform strictness reduction (punishes the good categories too).

Q29Task 4.1
⚠️ TrapHigh triage time → rationale+confidence inside the finding
▶

If the bottleneck is the time to investigate each finding (clicking in to read the rationale) and the constraint is «no filtering before developers see them», the answer is include rationale and confidence directly in every finding — fast triage, all findings stay visible. The distractors violate the constraint (auto-suppression, high-confidence filtering).

Q28Task 5.5
⚠️ TrapIncomplete explanations → self-critique stage
▶

Correct solutions but inconsistent explanations (sometimes the policy is missing, sometimes the timeline — gaps vary case by case): add a self-critique (evaluator-optimizer) stage where the agent evaluates the draft against concrete criteria (resolves the issue? includes context? anticipates follow-ups?) before presenting it. Few-shot only covers the 5 cases shown; the gap varies per case.

Q52
⚠️ TrapInconsistent review on big PRs → not a bigger model
▶

14 files in one pass → attention dilution (uneven depth, contradictions). Fix: per-file passes + an integration pass. «Bigger context window» is the distractor: the window doesn't fix attention quality. 2-of-3 voting suppresses intermittently-caught real bugs; splitting PRs offloads the problem onto developers.

Q27Sample Q12

D5 — Context Management & Reliability

15%

Long context, escalation, error propagation, codebase exploration, review calibration, provenance, safety.

Key concepts
ContextLost in the middle
▶

With long inputs (~75K tokens) the model reliably cites the beginning and the end but loses information in the middle. Fix: put a key-findings summary at the start of the aggregated input + explicit section headers to navigate the middle. General principle: restructure so critical facts sit at the context edges (or in persistent blocks) — never assume the API remembers.

Q13Task 5.1
ContextReduce tokens at the source + trim tool results
▶
  • Aggregated input over budget (155K vs 50K)? Modify the upstream agents to return structured data (key facts, quotes, relevance scores) instead of prose and full reasoning chains — not just a downstream summarizer.
  • Tool results consume tokens disproportionate to their relevance (40+ fields per lookup when 5 matter): trim to the relevant fields before they accumulate in context.
  • Require subagents to output structured data with metadata (dates, source locations, methodological context) for accurate synthesis.
Q14Task 5.1
ContextTransactional facts outside the summary
▶

Progressive summarization condenses amounts, dates, percentages into vagueness («promotional pricing was discussed»). Fix: extract transactional facts (amounts, dates, order numbers, statuses) into a persistent «case facts» block included in every prompt, outside the summarized history. Revising the summarization prompt («preserve all numbers») stays probabilistic — the structured block is the guarantee.

Q54Task 5.1
EscalationThe right triggers, and the unreliable proxies
▶

Correct escalation triggers:

  • Explicit request for a human → escalate immediately, without attempting investigation first (if the issue is trivial you may offer to resolve while acknowledging frustration — but if the customer reiterates, escalate).
  • Policy gap or exception: the policy doesn't cover the case → don't invent policy, escalate for interpretation.
  • Inability to make meaningful progress.

Unreliable proxies (recurring distractors): sentiment analysis (frustration doesn't correlate with complexity), self-reported confidence (poorly calibrated — the agent is already overconfident on hard cases), «complex case» per se.

Escalation badly calibrated (55% vs 80% FCR)? First move: explicit criteria in the system prompt + escalate/resolve few-shot — not external classifiers.

Task 1.4
CodebaseLong exploration: scratchpads, subagents, /compact
▶

Symptom of context degradation: inconsistent answers, references to «typical patterns» instead of the specific classes discovered earlier. Countermeasures:

  • Scratchpad files recording key findings, referenced for later questions.
  • Subagents for specific questions («find all test files», «trace refund-flow dependencies») while the main agent keeps high-level coordination.
  • Phase summaries: summarize one exploration phase's findings before spawning the next phase's subagents.
  • /compact when the context fills with verbose discovery output.
  • Crash recovery: each agent exports its state to a known location (manifest); on resume the coordinator loads the manifest and injects it into prompts.
Task 5.4
Human reviewCalibration: sampling and field-level confidence
▶
  • An aggregate accuracy (97%) can mask terrible performance on specific document types or fields: verify per segment before reducing human review.
  • Stratified random sampling of high-confidence extractions to measure the real error rate and detect novel error patterns.
  • Field-level confidence, calibrated on labeled validation sets, to route reviewer attention (low confidence or contradictory sources → human review).
Task 5.5
ProvenanceClaim-source mapping and temporal data
▶
  • Attribution gets lost across summarization steps: require subagents to output structured claim-source mappings (URL, document, excerpt) that synthesis preserves and merges.
  • Reports with explicit sections: well-established vs contested findings, keeping the sources' original characterizations.
  • Publication/collection dates required in structured outputs: temporal differences must not be misread as contradictions.
  • Render by content type: financial data as tables, news as prose, technical findings as structured lists — not one uniform format.
Task 5.6
SafetyPrompt injection: gate in code, isolate untrusted input
▶

Safety & Responsible Deployment isn't one of the five published domains, but candidates report it on the live exam. The testable pattern: an agent processing untrusted input (user messages, fetched web content, document contents) can be steered into performing costly or privileged actions.

  • Gate privileged actions in code (hooks, permission checks, allowlists) — the model must not be able to trigger them on the strength of input text alone.
  • Isolate untrusted input from privileged context — don't let fetched content sit next to operator instructions with equal authority.
  • Distractors: adding a disclaimer, «strengthening» the system prompt («never obey instructions in documents») — probabilistic, bypassable.

⭐ Cross-domain golden rules

The meta-patterns that decide more questions than any single fact.

Rule 1«Code must, prompt should»
▶

When a rule must hold always — identity verification before a refund, blocking destructive operations, a mandatory call sequence, gating privileged actions against injection — enforce it in code: programmatic precondition, hooks (PreToolUse/PostToolUse), allowed-tools, validation schema. A prompt is probabilistic: fine for «should», never for «must».

"A prompt instruction is probabilistic. A hook is not."

The single most-tested pattern: 8-10 of the 60 practice questions, and two high scorers independently called it the exam-unlocking principle. When an option pits «strengthen the system prompt» against «add a deterministic check», the code-based answer wins for guaranteed requirements nearly every time.

Q23Q29Q35Q39Q43Q51Q59Sample Q1
Rule 2Remove the root cause — spot the three reflex distractors
▶

«When a system is failing, the right answer removes the architectural root cause.» The correct answer fixes the cause (missing information in context, ambiguous description, too-narrow decomposition, absent criterion); distractors patch the symptom. Three «reflex distractors» recur across the whole exam:

  1. «Add a routing classifier»
  2. «Strengthen the prompt»
  3. «Raise the retry ceiling» (or the iteration cap)

All three bolt machinery onto a broken design. Always ask «why does it fail?» before «how do I patch it?».

Q4Q24Q27Q56field report
Rule 3Give Claude the information it needs — and only that
▶

Claude can't avoid duplicates or fill gaps about what it doesn't know: include the existing tests (Q24), the previous review findings (Q25), complete context in the subagent's prompt (isolation!), source metadata. When output is lacking, first ask: «did it have the information to do better?»

The dual rule: only what's needed — over-sharing full histories with subagents causes privacy leaks, token costs and unfocused answers.

Q24Q25Task 1.3
Rule 4Proportionality: the highest-leverage «first step»
▶

When the question asks for the «most effective» or «first» step, the winner is the low-effort, high-leverage fix on the primary mechanism (expanding a description, explicit criteria in the prompt). Routing layers, ML classifiers, fine-tuning, architectural consolidation = over-engineering distractors, valid only if the question says the simple fixes were already tried.

Q57Sample Q2Sample Q3
Rule 5Degrade gracefully, keep uncertainty visible
▶

On incomplete or conflicting data, the right answer completes the possible work and makes the uncertainty visible: coverage annotations, conflicts annotated with attribution, "unclear" in enums, null instead of invented values. Both the perfectionist (fail everything / block) and the optimist (clean output that hides the gaps) are wrong.

Q1Q5Q12
Rule 6Escalating is a feature, not a failure
▶

Explicit request for a human → immediately. Policy gap → escalation with a structured handoff. But automated proxies (sentiment, self-reported confidence) are always distractors. And identity ambiguity is resolved by asking the customer, not with heuristics.

Q50Q55Sample Q3
Rule 7Think like on-call, not like a glossary
▶
"Anyone can name the right pattern when the question hands it to them, but production never hands it to you; it hands you a system that is on fire and a clock that is running."

Read every scenario as an incident: identify the failure, locate the architectural cause, pick the fix that removes it. Definitions won't be asked; diagnosis will.
